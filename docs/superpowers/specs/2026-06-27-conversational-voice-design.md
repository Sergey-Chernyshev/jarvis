# Дизайн: разговорный голосовой ассистент Jarvis (Haiku-мозг)

Дата: 2026-06-27
Статус: на ревью
Под-проект 2 из дорожной карты «Jarvis как полноценный голосовой ассистент».
Строится поверх под-проекта 1 (голосовая маршрутизация, модуль `route/`).

## Контекст и место в дорожной карте

Под-проект 1 сделал: «Hey Jarvis» → STT → детерминированный роутер → stage-then-send
в нужную сессию, с HUD и индикатором «слышу». Маршрутизация была ЕДИНСТВЕННЫМ
исходом реплики.

Под-проект 2 превращает это в **разговорного ассистента**: Джарвис слушает
реплику и сам решает, что это — вопрос («сколько времени?», «что у frontend?»),
действие над инфраструктурой («переключи на opus», «не спи 2 часа») или промпт
для сессии. Отвечает **голосом**. Маршрутизация из под-проекта 1 становится
ОДНИМ из «скилов», которые мозг может вызвать.

Решения, согласованные на брейншторме:
- **Скоуп: всё.** Мозгу доступен весь набор скилов Джарвиса/инфры: чтение
  (sessions/chats/metrics/tasks/limits/time), маршрутизация, управление
  (model/effort/keep-awake/mute/settings).
- **Диалог: полный многоходовый** — непрерывное слушание после wake, VAD-
  эндпойнтинг, память разговора, барж-ин.
- **Модель: только Haiku**, через уже настроенный проксированный
  `claude_bin::run_haiku` (single-shot).
- **Конец разговора:** тишина ~8-10с ИЛИ стоп-фраза («спасибо/хватит»); барж-ин
  включён (речь прерывает озвучку Джарвиса).
- **Архитектура мозга: подход A** — Rust-оркестратор + структурный Haiku
  (один `run_haiku` на ход → JSON-план; Rust исполняет, гейтит, озвучивает).

## Что уже есть (переиспользуем)

- TTS: `Voice::speak_text(title, body, kind, toast_id)` / `test_phrase(text)` —
  Silero, русский, очередь + ducking + mute (`src-tauri/src/voice/`). Нужен
  небольшой `stop`/прерывание текущей озвучки для барж-ина (см. §Открытые вопросы).
- Haiku с проксей: `claude_bin::run_haiku(prompt, timeout) -> Option<String>` —
  хардненный single-shot (без MCP/хуков), прокси сохранена. Это и есть «мозг».
- Под-проект 1: `route::score` (скорер цели), `route::stage` (stage-then-send +
  держатель single-flight), `route::pick` (пикер), `route::hud` (фазы HUD),
  `route::classify` (узкий выбор) — переиспользуются как «route-скил».
- Сигнал тишины микрофона `AudioHub::is_mic_silent()` + событие `audio_state`
  (для VAD-старта и индикатора).
- Капабилити-реестр (`mcp__jarvis__*`) — список действий; в подходе A их вызывает
  Rust-диспатч, не Haiku напрямую (надёжнее на Haiku, чистый гейтинг).

## Архитектура (подход A): Rust-оркестратор + структурный Haiku

Мозг — НЕ MCP-агент-луп, а **планировщик на один ход**: Rust собирает контекст,
делает ОДИН `run_haiku` со структурированным выводом, парсит план и исполняет.
Это играет на сильную сторону Haiku (одно структурное решение, без хрупкого
многотул-протокола, где Haiku слабее всего), даёт Rust полный контроль над
безопасностью, HUD, TTS, памятью и латентностью, и полностью юнит-тестируемо.

### Поток (многоходовый)

```
«Hey Jarvis» → convo::start: открыть РАЗГОВОР (single-flight на весь разговор)
  loop turn:
    LISTEN (VAD-эндпойнтинг): ждём начало речи → пишем до трейлинг-тишины ~0.8с
       • барж-ин: если идёт TTS и началась речь → voice::stop → захват
       • нет начала речи ~8-10с → END (тишина)
    STT(pcm) → transcript
       • пусто → короткий ре-промпт «не расслышал», следующий ход
       • стоп-фраза → speak прощание, END
    PLAN: snapshot = build_snapshot(daemon); prompt = build_plan_prompt(snapshot, memory, transcript)
          plan = parse_plan(run_haiku(prompt))           // {speak, action?, end?, need_followup?}
    EXECUTE (если plan.action):
       read-скил   → данные из Rust (часто уже в снапшоте); при need_followup — 2-й run_haiku фразирует
       route-скил  → route::score + route::stage (окно отмены) / route::pick
       control-скил→ confirm/stage-окно → выполнить капабилити
    SPEAK plan.speak (Silero) + HUD: транскрипт + ответ
    memory.push(turn);  if plan.end → speak короткое прощание; END
  END: закрыть разговор, снять single-flight, HUD «разговор закрыт», вернуться к wake-only
```

### Снапшот мира (ключ к «один вызов на ход»)

`build_snapshot(daemon)` всегда кладётся в промпт, чтобы Haiku отвечал на
большинство read-вопросов без отдельного tool-вызова:
- локальные время и дата;
- живые сессии: для каждой `{короткий id, project, task, status, branch,
  last_prompt(сжат), в tmux ли}`; счётчики ждут/работают; есть ли лимит;
- состояние Джарвиса: mute, keep-awake (вкл/таймер).
Глубокие данные (полный чат сессии, детальные метрики) НЕ в снапшоте — их Haiku
запрашивает через `action` (read-скил), Rust достаёт, при нужде 2-й вызов фразит.
Бюджет: ≤2 вызова Haiku на ход.

## Компоненты (новый слой `convo/`, переиспользует `route/`)

- `convo/mod.rs` — оркестратор разговора: многоходовый цикл, состояние/память,
  условия конца, single-flight на весь разговор. Новая точка входа из `on_wake`
  (вместо прямого `route_transcript`).
- `convo/listen.rs` — VAD-эндпойнтинг: автомат «ожидание речи → запись → трейлинг-
  тишина → конец». Сигналы: начало речи (для барж-ина), конец реплики, тишина-
  таймаут. Старт — энергетический порог поверх кадров `AudioHub`.
- `convo/snapshot.rs` — **чистая** сборка компактного снапшота из `Vec<Session>` +
  времени + состояния (mute/keep-awake/limits).
- `convo/plan.rs` — **чистая**: `build_plan_prompt(snapshot, skills_menu, memory,
  transcript)` + `parse_plan(&str) -> Plan`. Терпимый парс (fence/конверт/мусор →
  безопасный фолбэк «переспроси»).
- `convo/skills.rs` — реестр скилов: для каждого `{name, when, args-schema,
  side_effect}`. `skills_menu()` → строка-меню для промпта; `dispatch(action,
  daemon) -> SkillOutcome` (read → данные; side-effect → через consent-гейт).
- `convo/reply.rs` — озвучка `plan.speak` через `d.voice` + `stop()` для барж-ина;
  отражение текста в HUD.
- `convo/memory.rs` — скользящее окно ходов `{user, assistant, action_result}`,
  ограничение по числу/токенам.
- Переиспользование: `route::{score, stage, pick, classify, hud}`. `route::hud`
  расширяется фазами разговора (listening/thinking/speaking/conversation-active).

`on_wake` (action.rs) меняется: вместо `route_transcript(...)` зовёт
`convo::start(daemon, preroll, guard)`. `route_transcript`/`stage_and_send`
остаются как реализация route-скила.

## Меню скилов

Чтение (без подтверждения):
- `sessions_status` — сводка (обычно уже в снапшоте; явный скил для «перечисли все»).
- `session_detail{id}` — статус/задача/прогресс конкретной сессии.
- `session_chat{id}` — последние сообщения сессии (`chats.read`).
- `metrics{scope}` — usage/метрики.
- `tasks{id}` — доска задач сессии.
- `limits` — статус лимита.
- `time` — локальные время/дата (Rust, без модели).

Маршрутизация (сайд-эффект → окно отмены/пикер):
- `route{session_hint, prompt}` → `route::score` резолвит цель (или пикер) →
  `route::stage` (stage-then-send, 5с отмена).

Управление (сайд-эффект → подтверждение/окно отмены):
- `set_model{id, model}`, `set_effort{id, level}` (`sessions.control`),
- `keep_awake{minutes|off}`, `mute{on|off}` (+ при нужде узкий `settings_set`).

Most ходов: либо без action (Haiku отвечает из снапшота), либо ровно один action.

## Модель безопасности (микрофон недоверенный)

- Reads — свободно, без подтверждения.
- Сайд-эффекты — ТОЛЬКО через до-исполнительный consent: route → stage-окно/пикер
  (как в под-проекте 1); control → stage-окно отмены ИЛИ confirm-карточка с
  точным описанием эффекта. `ConfirmPolicy::Never` на Control для голоса НЕ
  вводим — границей служит видимое окно отмены / тап.
- Валидация плана: `action.skill` обязан быть из меню; аргументы (id сессии,
  модель, минуты) провалидированы против реального состояния; иначе действие НЕ
  выполняется, мозг переспрашивает. Rust никогда не исполняет нелистовой скил.
- Анти-инъекция: транскрипт и данные снапшота/`chats.read` помечены как ДАННЫЕ,
  не команды маршрутизации; явная инструкция в промпте.
- Барж-ин/прерывание безопасны (только стоп TTS).

## Многоходовость — детали

- **VAD-эндпойнтинг** (`convo/listen.rs`): автомат на кадрах аудио — порог
  энергии для старта речи, трейлинг-тишина ~0.8с для конца реплики, общий
  таймаут ожидания начала ~8-10с → конец разговора. Полноценный Silero-VAD —
  возможное улучшение позже; старт на энергии.
- **Память** (`convo/memory.rs`): последние N ходов в промпт; ограничение по
  токенам, старые ходи сжимаются/выпадают.
- **Конец**: тишина-таймаут ИЛИ `plan.end` (Haiku распознал «спасибо/хватит»),
  плюс локальный детект стоп-фразы как страховка. Короткое голосовое прощание.
- **Барж-ин**: пока `d.voice` играет, `convo/listen.rs` детектит начало речи →
  `voice::stop()` (новый) обрывает текущую озвучку и чистит очередь → захват.
- **HUD-фазы** (расширяем `route::hud`): `listening`(с уровнем) / `thinking` /
  `speaking`(текст ответа) / действия (staged/picker) / индикатор «разговор
  активен»; конец — снять индикатор.

## Латентность (ход, Haiku)

VAD-захват (реалтайм + ~0.8с) + STT (~0.5-2с) + Haiku-план (хардненный, ~1-3с) +
TTS до первого звука (~0.5-1.5с) ≈ **3-7с/ход** с HUD «Думаю…». Снапшот в промпте
убирает лишние round-trip’ы на большинстве read-вопросов.

## Этапы реализации (TDD, инкрементально)

1. **Снапшот + планировщик** (`snapshot.rs`, `plan.rs`) — чистые, ядро решения.
2. **Диспатч скилов** (`skills.rs`): сперва reads (данные), затем сайд-эффекты на
   `route`/control через consent.
3. **TTS-ответ + HUD-фазы** (`reply.rs`): даже ОДНОХОДОВЫЙ «вопрос → голосовой
   ответ» уже полезен и поставляем.
4. **VAD-эндпойнтинг** (`listen.rs`) — замена фикс-окна.
5. **Многоходовый цикл + память + условия конца** (`convo/mod.rs`, `memory.rs`).
6. **Барж-ин** (`voice::stop` + детект начала речи во время TTS).

Каждый этап — рабочий коммит; после этапа 3 уже есть голосовой Q&A, после 5 —
полноценный разговор.

## Тестирование (TDD)

- `plan.rs` (чистая): промпт содержит снапшот, меню скилов, память, untrusted-
  маркер; `parse_plan` терпим к fence/конверту/мусору → безопасный фолбэк.
- `snapshot.rs` (чистая): корректная компактная сводка из набора сессий +
  счётчики/лимит/mute.
- `skills.rs`: read → данные; сайд-эффект → гейт-путь (мок ядра); неизвестный
  скил → отказ; валидация аргументов (несуществующий id, плохая модель).
- `listen.rs`: автомат endpointing на синтетических кадрах энергии (старт/конец/
  таймаут/барж-ин-сигнал).
- `memory.rs`: ограничение окна по числу/токенам; порядок ходов.
- Оркестратор: последовательность ходов, условия конца, переход read→re-plan
  (с фейковым Haiku и ядром).
- Безопасность: сайд-эффект недостижим без consent; нелистовой скил отвергнут;
  транскрипт/данные трактуются как данные.

## Вне scope (возможные будущие под-проекты)

Полноценный Silero-VAD (если энергетического мало); создание НОВОЙ сессии
голосом (под-проект 3); проактивные голосовые уведомления-диалоги; не-Haiku
эскалация; персистентная долгая память между разговорами.

## Открытые вопросы для реализации

1. `voice::stop()` для барж-ина: у `Voice` есть mute/queue/dispose — нужно
   добавить «оборвать текущую озвучку + очистить очередь». Проверить
   `voice/player.rs`/`queue.rs`, как корректно прервать сайдкар-воспроизведение.
2. Порог энергии VAD и трейлинг-тишина: подобрать дефолты, выставить настройками;
   не путать с фикс-окном из под-проекта 1 (его заменяем для разговора).
3. Формат памяти в промпте при только-Haiku: держать компактно (Haiku-контекст
   меньше) — сколько ходов/токенов; стратегия сжатия.
