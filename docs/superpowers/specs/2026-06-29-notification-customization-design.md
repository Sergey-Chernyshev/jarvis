# Кастомизация уведомлений + Bluetooth-гейт озвучки

Дата: 2026-06-29
Статус: дизайн одобрен (макет прокликан), готов к плану
Issue: #24 (показывать текущую ветку)

## Цель

1. Показывать в карточке/тосте сессии **текущую ветку** (issue #24) и сделать
   содержимое уведомления **настраиваемым** (ветка/модель/усилие/токены/время).
2. **Настройки → Уведомления**: обширная панель с живым превью.
3. **Онбординг**: короткий шаг выбора (ветка + модель) после установки/обновления.
4. **Настройки → Голос**: озвучка только при подключённой **Bluetooth-гарнитуре**
   (вкл по умолчанию). Речь всегда = **проект + описание результата** (поля
   карточки в речь НЕ идут).

## Данные

Ветка уже есть: `model.rs Session.branch: Option<String>` (демон заполняет из
gitBranch). Модель — `Session.model`. Усилие/токены — из usage/effort.

### Новые поля settings.json (все additive, с дефолтами — миграция не нужна)

```jsonc
{
  "notify": {
    "content": { "branch": true, "model": false, "effort": false, "tokens": false, "time": false },
    "events":  { "done": true, "waiting": true, "limit": true },   // = бывшие notifyDone/notifyWaiting (+limit)
    "ttlSec": 8                                                     // 0 = не прятать
    // position остаётся верхнеуровневым (как сейчас)
  },
  "voice": { "bluetoothOnly": true }                               // новый тумблер
}
```

Совместимость: старые `notifyDone`/`notifyWaiting` читаем как фолбэк для
`notify.events.done`/`.waiting`, если `notify` отсутствует.

## Карточка/тост: мета-строка

Демон при сборке уведомления (daemon.rs, где формируется payload тоста, ~417-457)
добавляет поле `meta` — массив готовых сегментов по `notify.content` + данным
сессии:

- `branch` → `{ kind:"br", text:"⎇ <branch>" }`
- `model` → `{ kind:"md", text:"<model>" }`
- `effort` → `{ kind:"ef", text:"<low|high|max>" }`
- `tokens` → `{ kind:"plain", text:"<n>k tok" }`
- `time` → `{ kind:"plain", text:"HH:MM" }`

Пустые/отсутствующие сегменты опускаются. `toast.js` рендерит `meta` строкой под
заголовком (классы `.tmeta .br/.md/.ef`, разделитель `·`) — как в макете;
если `meta` пуст, строки нет (текущий минимальный вид). Главная панель
(`renderer.js`, карточка сессии) рендерит ту же мету из `Session` + настроек.

Заголовок остаётся `проект`; ветка/мод= в мете (вариант B из обсуждения —
ветка отдельным сегментом, не обрезается в заголовке).

## Настройки → Уведомления (renderNotify, settings2.js)

Панель `notify` (заменяет текущую мини-версию):
- **Превью** сверху: живая карточка-тост + переключатель статуса
  (Готово/Ждёт/Лимит) — меняет точку и пример текста.
- **Содержимое карточки**: 5 тумблеров (Ветка/Модель/Усилие/Токены/Время) →
  `notify.content.*`; превью обновляется вживую.
- **Уведомлять о**: Завершена/Ждёт ответа/Лимит-ошибка → `notify.events.*`.
- **Вид и поведение**: Позиция (По центру/В углу → `position`), Автоскрытие
  (5с/8с/Не прятать → `notify.ttlSec`), Озвучивать голосом (→ ссылка-подсказка
  на «Голос»; сам тумблер озвучки уже в «Голос»).

Каждое изменение → `window.jarvis.setSettings({...})` (как существующие).

## Онбординг (короткий шаг)

Экран «Что показывать в уведомлениях»: превью + 2 тумблера (Текущая ветка вкл,
Модель выкл) → пишет `notify.content.{branch,model}`. Подпись «остальное — в
Настройках». Показывается в существующем онбординг-флоу после установки/обновления
(если шаг ещё не пройден — флаг `notify.onboarded`).

## Голос: Bluetooth-гейт

- Новый тумблер в `renderVoice` (settings2.js): «Озвучивать только при
  Bluetooth-гарнитуре» → `voice.bluetoothOnly` (дефолт true).
- Бэкенд: `macos::bluetooth_audio_output_connected() -> bool` — подключено ли
  Bluetooth-аудиоустройство вывода. Реализация: CoreAudio — транспорт дефолтного
  output-устройства == Bluetooth (быстро, точно); при сложности — фолбэк через
  `system_profiler SPBluetoothDataType -json` с кэшем ~10с. Результат кэшируется
  (не дёргать на каждый тост).
- Гейт в `voice/mod.rs::speak_text`: если `voice.bluetoothOnly` и НЕ подключена
  BT-гарнитура → не озвучивать (тихо выходим, как при mute).
- **Содержимое речи не меняется**: всегда `проект + описание результата`
  (текущая логика speak_text), независимо от `notify.content`.

## Совместимость / «не сломать»

- Все новые поля — с дефолтами; загрузка мержит дефолты (settings.rs) → старые
  файлы читаются. `schemaVersion` не трогаем (изменения additive).
- Старые `notifyDone/notifyWaiting` уважаем как фолбэк.
- Если детект BT недоступен/ошибка → считаем «не подключено» (с `bluetoothOnly`
  это значит тишина) — НО дефолт можно сделать «при ошибке озвучивать», чтобы не
  терять уведомления молча. Решение: при ошибке детекта — НЕ глушить (fail-open),
  чтобы баг детекта не убивал озвучку.

## Тестирование

- Rust: построение `meta` по настройкам (юнит, чистая функция); фолбэк
  notifyDone→events; гейт speak_text (BT on/off × bluetoothOnly on/off, fail-open).
- JS: ручная проверка панели/онбординга (живое превью), node --check.
- Живая: ветка видна в тосте; тумблеры пишут settings; озвучка молчит без BT и
  говорит с BT; речь = проект+результат.
