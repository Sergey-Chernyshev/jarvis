# Сводка чата v2 + документы: возврат саммари и ревью доков внутри Jarvis

Дата: 2026-07-18 · Статус: дизайн по прямому запросу владельца · Предыстория:
chatsum v1 (PR #48) удалён hardening-спекой 10.07 (§4.1) как «неверно понятое
представление». **Решение отменено владельцем 18.07.2026**: режим «Сводка»
нужен, и он — точка входа в новый слой «Документы» (просмотр/ревью файлов,
которые агент создал/правил, прямо в Jarvis).

## 1. Продуктовая рамка

Jarvis — монитор/пульт. Пользователь глазами следит, что агент наделал, и
принимает решения. Два разрыва сегодня:

1. Лента чата сырая — «что по сути произошло за ход» не видно (это решал
   chatsum v1 — возвращаем).
2. Главный артефакт хода — часто **документ** (спека, план, README): агент его
   создал или отредактировал, и человеку нужно его *посмотреть/отревьюить*,
   не уходя из Jarvis. Сегодня для этого надо идти в редактор/гит руками.

Целевой флоу: ход завершился → карточка сводки говорит «правил
`docs/…-design.md`» → клик → документ открывается **в панели Jarvis**: либо
текущая версия (создан новый), либо «Изменения» (правил существующий) — и
кнопка «Открыть в редакторе», если нужно больше.

## 2. Что возвращаем как было (chatsum v1, спека 2026-07-03)

Движок и UX v1 переносятся без переосмысления (они были ок):

- Сегментация ходов, `TurnFacts` (файлы/команды из tool_use — детерминированно),
  LLM-проза поверх фактов, валидация путей ⊆ факты, кэш
  `~/.jarvis/turn-summaries/<sid>.json` с PROMPT_VERSION, ключ хода = ts.
- Карточки ходов: проза, файл-чипы, строка команд, «развернуть» в сырьё.
- Тумблер «Сводка / Лента» в шапке чата (localStorage, по умолчанию «Сводка»).
- Гейты токенов: on-Stop только для открытого чата, backfill 5 ходов, кнопка
  «Сводка» на старых.
- ipc `chat_summarize` + `file_open` (клик — открыть, ⌥ — Finder).

Порт на новый master обязан учесть: daemon.rs переписан хардингом (#53),
`ui/chat-contract.test.mjs` инвертируется (теперь стережёт НАЛИЧИЕ поверхности:
`chatModeSeg`, карточка, тумблер), память
`jarvis-chat-turn-summaries` обновить.

## 3. Новое: слой «Документы»

### 3.1 Просмотр документа в Jarvis (инкремент 2)

- Файл-чип с расширением `.md`/`.markdown` (и кнопка «Открыть документ» на
  карточке, если ход трогал доки) открывает **вьюер в панели**: слайд-овер
  поверх чата (паттерн qview/varPanel), заголовок = имя файла, тело =
  отрендеренный markdown.
- Рендер — свой микро-модуль `ui/markdown.js` (UMD, node-тесты): заголовки,
  списки, код/фенсы, **жирный**/*курсив*, ссылки, таблицы простые,
  цитаты. Всё экранируется (никакого сырого HTML из файла — содержимое
  недоверенное); ссылки наружу — только через `open` по клику, `javascript:` и
  прочее режется. Не тянем стороннюю либу: UI ванильный, безопасность важнее
  полноты CommonMark.
- Не-markdown файлы: v1 — моно-текст с подсветкой «как есть» (без раскраски),
  большие (>512КБ) — голова+хвост с пометкой.
- «Открыть в редакторе»: кнопка во вьюере и ⌥-меню чипа. v1 — системное
  `open <path>` (дефолт приложения macOS) + `open -R` (Finder, как было).
  Настройка «Редактор документов» (кастомная команда) — отложена до запроса.
- Источник путей — только валидированные факты хода (как в v1); вьюер читает
  файл через ipc `file_read` (новая команда: канонизация пути, лимит размера,
  UTF-8 lossy). Панель — полное доверие, но путь всё равно сверяем со списком
  фактов сессии, чтобы чип не мог открыть произвольный путь после инъекции в
  транскрипт.

### 3.2 Ревью изменений (инкремент 3)

Пользователю нужно видеть «что изменилось», когда агент правил существующий
док (архитектурный документ и т.п.):

- Вьюер получает **табы «Изменения» / «Документ»**. «Изменения» доступен, если
  файл лежит в git-репо сессии (cwd) и дифф не пуст.
- База сравнения v1 — git, точных снапшотов «до хода» не храним (честно и
  дёшево):
  1) есть незакоммиченные правки файла → `git diff HEAD -- <file>` (главный
     кейс: агент наделал, человек ревьюит до коммита);
  2) рабочее дерево по файлу чистое → дифф последнего коммита, тронувшего файл
     (`git log -1 -p --follow -- <file>`), с подписью «коммит <short> · <субъект>»;
  3) не в git → таба нет, только «Документ».
- Отображение: unified inline (удалённое красным зачёркнутым, добавленное
  зелёным), **по параграфам**: смежные изменённые строки схлопываются в блок;
  между блоками — свёрнутый контекст («⋯ 42 строки без изменений»).
- Реализация диффа: Rust-сторона (ipc `file_diff`) отдаёт готовые ханки
  (спавн `git`, парс unified), UI только красит. Никакого diff-алгоритма в JS.
- **Пометки «что изменилось» (инкремент 4, on-demand)**: кнопка «Пояснить» на
  блоке → служебный LLM (тот же `run_service_llm`) получает ханк + 1-2 абзаца
  контекста, возвращает 1 предложение по-русски; кэш рядом с turn-summaries.
  Не автоматом: служебный LLM медленный (~15с) и это токены.

### 3.3 Сводка знает про доки

- В карточке хода doc-файлы (`docs/**`, `*.md`) выделяются: чип с 📄 идёт
  первым и получает бейдж `изменён`/`создан` (из tool_use: Write без
  предшествующего чтения ≈ создан; Edit ≈ изменён — эвристика фактов, не LLM).
- Если ход правил ≥1 док, карточка получает кнопку «Открыть документ» (первый
  док) — это главный CTA сценария «продевьюить, что агент написал».

## 4. Безопасность

- Содержимое файлов и транскрипта — недоверенное: рендер только с полным
  экранированием, пути только из фактов, `file_read`/`file_diff` — канонизация
  + отказ вне списка фактов сессии.
- `file_diff` спавнит `git` строго с `-C <cwd сессии>` и `--` перед путём
  (никакой интерпретации файла как ревизии/флага).
- LLM-выход (проза, пояснения ханков) — только текст, рендер экранированный;
  ссылки/пути из LLM не кликабельны (как в v1).

1. **Возврат chatsum** — revert-of-revert с портом на новый master (daemon.rs
   после #53, renderer.js после #56/#58), инверсия контракт-теста, зелёные

2. **Вьюер документов**: `ui/markdown.js` + слайд-овер + `file_read` +

3. **Дифф-ревью**: `file_diff` (git), табы, unified-по-параграфам.
4. **Пояснения к ханкам** (LLM, on-demand) — отдельно, после обкатки 2-3.

Каждый — отдельный PR с тестами; UI-логика — чистыми UMD-модулями под
node-тесты (паттерн `ui/onboarding-state.js`).

## 6. Тестирование

- Rust: порт v1-тестов turns/turnsum; `file_read` (канонизация, лимит, отказ

  переименование, бинарь → отказ).
- UI (node): `markdown.js` (рендер + экранирование + резка `javascript:`),

- Живое: тумблер, схлопывание хода по Stop, клик-чип → вьюер, «Изменения» на
  реально правленном доке, «Открыть в редакторе».
